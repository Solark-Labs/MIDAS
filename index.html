HOW MIDAS WORKS
Metadata broker architecture · zero PII · 6-step flow · auto-settlement
WHAT MIDAS IS
A neutral metadata broker receiving anonymized destination geo-hash signals from competing platforms, detecting when orders from different platforms target the same geo-hash zone, firing consolidation signals back, and auto-settling cost splits. Never stores customer PII. Never manages riders directly.
WHAT MIDAS IS NOT
Not a logistics company. Not an employer. Not a data aggregator. Does not see competitor pricing, inventory, or customer identities. Does not assign riders (each platform dispatches its own). Does not store full delivery addresses — only 6-char geo-hash cells.
STEP 01 — ORDER PLACED
Customer orders on Zepto. Zepto selects nearest hub. If multi-hub, Zepto handles split internally. MIDAS not involved.
// Zepto internal — MIDAS sees nothing yet hub = nearest_hub_with_stock(order)
STEP 02 — METADATA PACKET
After order confirmed, Zepto sends MIDAS an anonymized 6-field packet. No customer data. Only logistics signals.
{plat:"ZPT", hub:"ZPT_HYD_KOND", dest_geo:"GH7F4X", weight_kg:3.2, sla_tier:"T1", order_token:"ANON_8f3a"}
STEP 03 — GEO-HASH MATCH
MIDAS checks pool for same geo-hash. If SWG order for "GH7F4X" exists and hubs are ≤800m — match fires in <2ms.
pool["GH7F4X"] → SWG(2.8kg)+ZPT(3.2kg)=6.0kg ✓ hub_relay_dist = 220m ✓ → MATCH in 1.4ms
STEP 04 — CONSOLIDATION SIGNAL
MIDAS → Swiggy dispatch API: "Your rider can relay ZPT parcel from ZPT hub (220m detour) to geo GH7F4X." Swiggy decides and dispatches.
{action:"CONSOLIDATION_OPPORTUNITY", relay_hub:"ZPT_HYD_KOND", relay_geo:"GH7F4X", span_id:"SPAN_SWG_R012_ZPT_R044_ORD88812"}
STEP 05 — QR RELAY & HANDOVER
Rider A stops at ZPT hub (220m detour). Rider B scans QR. 3 simultaneous checks pass. Atomic DB write transfers custody. Rider B receives GPS + masked phone only.
SPAN_SWG_R012_ZPT_R044_ORD88812 → checks(a,b,c) PASS ATOMIC: remove SWG_R012 → add ZPT_R044
STEP 06 — AUTO-SETTLEMENT
Delivery confirmed. Session erased. MIDAS calculates weight-proportional split. ZPT debited ₹14.20. SWG credited ₹12.00. MIDAS takes ₹3.70 broker fee. Immutable ledger.
ZPT pays: ₹14.20 · SWG earns: ₹12.00 · MIDAS: ₹3.70 InvoiceID: INV-ZPT-00412 → bound to TraceID
SCALE & HUBS
8 cities · 100+ hubs · geo-hash federated · city-native deployment
India-Scale Architecture: Each city gets an independent MIDAS node. Zero data between cities. Hub zones use 6-char geo-hash cells (Geohash-6, ~1.2km precision). Delivery boys are hub-registered. MIDAS only pairs orders from hubs whose geo-hash cells are identical (same 1.2km delivery cluster). No cross-hub, no cross-city mixing. Adding a city = deploying a new node + uploading that city's hub geo-hash map.

Why SLA Breaches Still Happen (And Why That's OK)
In LATENCY mode, MIDAS assigns the nearest available agent in <200ms — preventing further delay. SLA breaches during CONSOLIDATION mode indicate a hub-capacity issue, not a MIDAS failure. Every breach is recorded with the stress mode active at that time, creating a forensic audit trail that proves MIDAS reduced SLA impact vs no-MIDAS scenario.
