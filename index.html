          💡 How Fractional Settlement Works inside the Beckn Protocol:

          Standard ONDC payouts require direct buyer-to-single-rider settlements. Under the MIDAS multi-hop relay model, the matched transaction fee is fractionally distributed:
          
    
            
  • First-Mile Runner: Gets ₹12-15 for the short neighborhood collection (Seller ➔ Relay Node).
    • 
            
  • Primary Last-Mile Rider: Receives the main bulk payout (Relay Node ➔ Buyer) dynamically accumulated across multiple consolidated orders.
    • 
            
  • The buyer's app automatically splits the payment at the ONDC settlement registry, ensuring instantaneous rider payout clearance.
    • 
          

        

        
        

          💳 The 3-Tier Checkout & Quoting Protocol:

          MIDAS matches orders on the fly without freezes using background Redis caching and a 3-tier customer experience model:
          
    
            
  • Tier 1 (Instant Pre-Pay Hold): User waits 60s at checkout. If matches consolidate within the hold buffer, they pay the exact low rate immediately.
    • 
            
  • Tier 2 (Prep-Window Matching): Standard checkout upfront. While the merchant prepares the order (cooking ETA), MIDAS runs background calculations, disclosing the final consolidated fee mid-transit.
    • 
            
  • Tier 3 (Post-Checkout Cashback Fallback): Dispatches standard. If a route consolidates later in-transit (street-level swaps), the user is refunded a surprise cashback.

1. Title of Invention

            

              MULTI-PLATFORM DELIVERY ORDER CONSOLIDATION SYSTEM WITH STRUCTURED TRACE IDENTIFIERS, RETURN-ORDER MATCHING, AND QR-VERIFIED INTER-CARRIER HANDOVER
            

          


          
          

            
2. Abstract

            

              A computer-implemented middleware system that sits between multiple competing e-commerce delivery platforms and their delivery networks. When an order arrives, the system reads the platform code, creation time, destination zone, and cart weight supplied by the platform and assembles them into a single structured identifier called a Trace ID. The system then assigns a processing label to that identifier — such as HEAVY_WEIGHT, INTRA_CONSOLIDATED, INTER_CONSOLIDATED, RETURN_PENDING, SPAN_DETECTED, or DIRECT_ASSIGNED — based solely on reading the identifier string, without querying a database. Each label determines the processing path: heavy orders go directly to dispatch; eligible orders are matched by weight with other orders going to the same zone, within the same platform first and across platforms second; a recently dispatched partner still near the hub is notified if a same-zone order arrives shortly after dispatch; customers may add forgotten items before packing starts, paying a fee that grows with time; return orders from completed deliveries are held in a dedicated section and matched against outbound orders going to the same origin zone within a set time window, eliminating separate return trips; when an order must change hands between partners from different platforms, a handover identifier called a SPAN_ID is created, and the receiving partner receives only the delivery address and a masked phone number through a session that ends when delivery is confirmed. Every order is guaranteed to reach a dispatched state. No customer personal data crosses platform boundaries at any point.
            

          


          
          

            
3. Field of the Invention

            

              This invention relates to a computer-implemented logistics coordination system that manages delivery orders across multiple independent e-commerce platforms. The system generates structured order identifiers, assigns processing labels based on those identifiers, matches outbound and return orders for combined trips, notifies recently dispatched partners of late same-zone orders, supports controlled order changes before dispatch, handles authenticated transfers between partners from different platforms, and keeps customer personal data isolated throughout.
            

          


          
          

            
4. Background and Technical Problem Statement

            

              
Problem 1 — Duplicate route coverage: Partners from competing platforms travel to the same delivery zone simultaneously with no mutual awareness. This creates massive physical congestion and duplicates road coverage.

              
Problem 2 — Underused vehicle capacity: Riders carry only their own platform's orders, often a single small item. No cross-platform pairing by weight exists to maximize physical capacity before dispatch.

              
Problem 3 — No fast cross-platform order identification: Standard order IDs carry no structural metadata. Finding consolidation opportunities requires expensive, database-heavy joins.

              
Problem 4 — Empty return trips: After completing final-mile deliveries, riders return to the origin hub empty, resulting in wasted fuel and mileage.

              
Problem 5 — No way to reach a dispatched partner for a late same-zone order: Once a rider is dispatched, same-zone orders arriving moments later generate a second independent trip, rather than modifying the active rider's route.

              
Problem 6 — No order modification after placement: Forgotten items generate a separate order and redundant logistics leg.

              
Problem 7 — No authenticated inter-carrier order transfer: Competing networks lack a secure, fraud-resistant protocol for riders to exchange packages on the street.

              
Problem 8 — Customer data exposure: Relay handovers expose full personal details (PII) to competing platform riders.

            

          


          
          

            
5. Objects of the Invention

            

              The primary objectives are: to encode platform, time, zone, and weight into a single string identifier (Trace ID) to run matching operations without database lookups; to route heavy orders directly; to match outbound/return orders to eliminate empty mileage; to send hold notifications to recently dispatched nearby riders; to issue secure handover SPAN_IDs verified via QR code scan; and to isolate customer PII compliant with the Digital Personal Data Protection Act, 2023.
            

          


          
          

            
6. Detailed Description of the Invention

            

              
Structured Order Identifier — Trace ID Format:

                TRACE_ID = [PLATFORM_CODE]_[HHMMSS]_[ZONE_CODE]_[WEIGHT_KG]

                Example: SHF_123045_500081_03.50 (Shadowfax, 12:30:45 PM, geohash 500081, 3.50 kg).
              

              
Return Order format:

                RETURN_TRACE_ID = RTN_[PLATFORM_CODE]_[HHMMSS]_[ORIGIN_ZONE]_[WEIGHT_KG]
              

              
Step-by-Step Processing Pipeline:

                - Step 1: Order Intake & Modification Window: Holds orders for 60-120 seconds for potential item additions.

- Step 2: Weight Check: If weight exceeds MAX_TRIP_WEIGHT (default 45 kg), bypasses matching and dispatches immediately as `HEAVY_WEIGHT`.

              
- Step 3: Parallel Order Matching: Checks same-platform and cross-platform orders for weight compatibility and arrival time to resolve ties. Creates an `INTRA_CONSOLIDATED` or `INTER_CONSOLIDATED` log.

              
- Step 4: Return Order Matching: Creates a `RETURN_TRACE_ID` when a rider completes delivery, matching them with outbound orders back to their origin zone to eliminate empty trips.

              
- Step 5: Predictive Transaction Hold: Holds order for up to 90 seconds if checkout activity indicates a neighboring buyer is currently processing a payment.

              
- Step 6: Forced Proximity Hold: Holds a confirmed order at the hub for 60 seconds if a dispatched rider is still within 500-1000m and has weight capacity.

              
- Step 7: Handover Identifier and QR Verification: Generates a `SPAN_ID` for street swaps:
                SPAN_[ORIGIN_PARTNER_ID]_[RELAY_PARTNER_ID]_[ORDER_REFERENCE]_[TIMESTAMP]
                Ownership is transferred via single database transaction upon scanning the packet QR code.
              

              
- Step 8: Direct Assignment Fallback: If no match is found, dispatches as standard `DIRECT_ASSIGNED`.

            

          


          
          

            
7. How Consolidation Happens (MIDAS Algorithms)

            

              MIDAS reads the structured metadata in incoming order headers. Instead of running expensive spatial queries on database tables, it maps the destination geohash zone. As orders are logged, the gateway checks for overlaps in active zones. 
              If two orders originate in the same geohash cluster (or neighboring 2km sectors) and fall within the same 10-15 minute time window:
              


              Consolidated Run Execution: Rather than dispatching Rider 1 from Seller 1 and Rider 2 from Seller 2 separately, the system selects a single active rider. The rider visits Seller 1, sweeps to Seller 2, packs both items, and drops them sequentially to Buyer 1 and Buyer 2. 
            

          


          
          

            
8. Delivery Fee Calculation & Payout Splits

            

              MIDAS enforces a strictly transparent pricing structure that replaces arbitrary delivery pricing:
              


              1. Standard Delivery Fee: calculated based on distance: \(Fee = ₹35.00 + (₹4.00 \times Distance\_in\_km)\).
              

              2. Consolidated MIDAS Fee: when route overlap is established, the delivery fee for both buyers is consolidated. The consolidated fee is priced at 45% of the combined standard fee (yielding a 55% direct upfront saving for each buyer).
              

              3. Fractional Settlement Splits: The consolidated fee is split dynamically:
              

              - Platform Gateway: receives 8% of the transaction fee.
              

              - Runner (Proximity Swaps): receives 32% if a street relay swap occurs.
              

              - Last-mile Rider: receives the remaining balance (92% if direct multi-stop, 60% if street swap relay).
